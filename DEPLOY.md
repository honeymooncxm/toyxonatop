# Запуск To'yxonaTop в реальную эксплуатацию

## Как это устроено сейчас

Сейчас сайт работает **только у вас на компьютере**: команда `npm run dev` запускает
временный сервер на `localhost:3000`, а все данные (залы, пользователи, отзывы)
хранятся в одном файле `dev.db` прямо в папке проекта. Это нормально для разработки
и демонстрации, но:

- этот сервер выключается, как только закрывается терминал/компьютер;
- к нему не может зайти никто, кроме вас;
- файловая база SQLite не подходит для реального хостинга (Vercel и подобные
  платформы не хранят файлы между запросами).

Чтобы сайтом могли пользоваться реальные люди — нужен: **хостинг** (кто-то, кто
держит сайт включённым 24/7 и даёт ему адрес в интернете) и **настоящая база
данных** в облаке.

## Что понадобится (все варианты ниже — с бесплatным тарифом для старта)

| Что | Зачем | Где взять |
|---|---|---|
| GitHub | Хранить код проекта | github.com |
| Vercel | Хостинг сайта (создатели Next.js, идеальная связка) | vercel.com |
| PostgreSQL база | Хранить реальные данные (залы, пользователи, отзывы) | neon.tech (проще всего) / supabase.com / vercel.com/storage |
| Cloudinary | Хранить настоящие фото залов, которые будут загружать владельцы | cloudinary.com |
| Домен | Красивый адрес вида toyxonatop.uz | любой регистратор доменов (напр. reg.uz) |

Аккаунты на этих сервисах должны создать **вы сами** через браузер (я не могу
создавать их за вас) — это буквально форма регистрации на 2 минуты каждая, без
привязки карты для бесплатного тарифа. Дальше присылаете мне нужные ключи/адреса,
и я подключу всё к коду.

## Что я уже подготовил в коде

- **`prisma/seed.production.ts`** — отдельный скрипт заполнения базы для боевого
  запуска. В отличие от обычного `prisma/seed.ts` (который создаёт 24 демо-зала,
  тестовых владельцев и отзывы для показа), этот скрипт создаёт **только**:
  список услуг/категорий (нужны для фильтров и формы регистрации зала) и **один
  ваш настоящий admin-аккаунт**. Ни одного фейкового зала. Скрипт сам откажется
  запускаться, если пароль администратора не задан или оставлен по умолчанию.
- **`.env.production.example`** — шаблон переменных окружения для боевого сервера.
- Схема базы данных (`prisma/schema.prisma`) изначально написана так, чтобы
  переключение с SQLite на PostgreSQL требовало минимум правок (см. Шаг 3).

## Пошаговая инструкция

### Шаг 1. Код на GitHub

```bash
git init
git add .
git commit -m "Initial commit"
```

Создайте пустой репозиторий на github.com, затем:

```bash
git remote add origin https://github.com/ВАШ_АККАУНТ/toyxonatop.git
git push -u origin main
```

### Шаг 2. База данных PostgreSQL (рекомендую Neon)

1. Зарегистрируйтесь на neon.tech, создайте новый проект.
2. Скопируйте строку подключения (Connection string) — она выглядит как
   `postgresql://user:password@ep-xxx.neon.tech/dbname?sslmode=require`.
3. Пришлите её мне (или сохраните для Шага 4) — она понадобится как `DATABASE_URL`.

### Шаг 3. Переключение схемы на PostgreSQL

Это единственное место, где нужно менять сам код при переходе с локальной SQLite
на реальную базу — сообщите мне, когда получите строку подключения из Шага 2, и я:

1. Поменяю `provider = "sqlite"` на `provider = "postgresql"` в
   `prisma/schema.prisma`;
2. Обновлю `lib/prisma.ts` и `prisma/seed.production.ts` на PostgreSQL-адаптер
   (`@prisma/adapter-pg` вместо `@prisma/adapter-better-sqlite3`);
3. Создам чистую PostgreSQL-миграцию (`npx prisma migrate dev --name init`) —
   старые миграции были написаны под SQLite и не подходят;
4. Прогоню `npx prisma migrate deploy` на вашей боевой базе.

### Шаг 4. Деплой на Vercel

1. На vercel.com → "Add New Project" → выберите репозиторий из Шага 1.
2. В "Environment Variables" добавьте все переменные из `.env.production.example`
   (реальные значения: `DATABASE_URL` из Шага 2, свой `JWT_SECRET` — сгенерировать
   можно командой `openssl rand -base64 48`, свой `ADMIN_EMAIL`/`ADMIN_PASSWORD`).
3. Нажмите Deploy. Через минуту сайт будет доступен по адресу вида
   `toyxonatop.vercel.app`.

### Шаг 5. Заполнение базы начальными данными

Один раз после первого деплоя (локально, указывая на боевую базу):

```bash
DATABASE_URL="ваша-строка-из-neon" ADMIN_EMAIL="admin@..." ADMIN_PASSWORD="..." npx tsx prisma/seed.production.ts
```

Это создаст услуги/категории и ваш админ-аккаунт. Дальше реальные владельцы залов
регистрируются сами через сайт, а вы одобряете их через `/admin`.

### Шаг 6. Фото залов (Cloudinary)

Без этого шага форма регистрации зала всё равно работает — просто владельцы
вставляют ссылки на фото вручную. Чтобы дать им нормальную загрузку файлов:

1. Зарегистрируйтесь на cloudinary.com (бесплатный тариф).
2. Скопируйте Cloud name, API Key, API Secret из Dashboard.
3. Добавьте их в переменные окружения на Vercel.

### Шаг 7. Свой домен

В Vercel: Project → Settings → Domains → добавить `toyxonatop.uz` (или что купите
у регистратора) и прописать у регистратора DNS-запись, которую покажет Vercel.

## Чек-лист безопасности перед запуском

- [ ] `ADMIN_PASSWORD` — надёжный, не `Admin123!`
- [ ] `JWT_SECRET` — длинная случайная строка, не тестовое значение из репозитория
- [ ] `.env` с реальными ключами **не закоммичен** в git (уже в `.gitignore`)
- [ ] База заполнена через `seed.production.ts`, а не обычный демо-`seed.ts`
- [ ] Прогнали чек регистрации/входа/подачи зала на реальном адресе после деплоя
